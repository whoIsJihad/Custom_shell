# Step 10: Signal Handling

A robust shell needs to handle asynchronous events, such as user interruptions like `Ctrl+C`. This final step provides a primer on signal management.

## What are Signals?

Signals are a fundamental mechanism in Unix-like systems for handling asynchronous events. They serve as a form of lightweight, software-level interrupt, notifying a process that a specific event has occurred. Each signal is identified by a unique integer and a symbolic name (e.g., `SIGINT` for signal 2) and is generated in response to various conditions, such as a hardware exception, a kernel notification, or a user action.

For an interactive shell, the most relevant signals are those generated by keyboard input:

*   **`SIGINT` (Signal Interrupt)**: This signal is typically sent to the foreground process when the user presses `Ctrl+C`. The default action for a process receiving `SIGINT` is to terminate immediately.
*   **`SIGTSTP` (Signal Terminal Stop)**: This signal is sent when the user presses `Ctrl+Z`. The default action is to suspend (pause) the process's execution.
*   **`SIGCHLD` (Signal Child)**: This signal is sent from the kernel to a parent process whenever one of its child processes terminates, stops, or is continued. It serves as an asynchronous notification that `waitpid()` can be called to reap the child.

## Signal Dispositions

When a process receives a signal, it can respond in one of three ways, known as the signal's **disposition**:

1.  **Perform the Default Action**: The process can allow the kernel to perform the default behavior associated with the signal (e.g., terminate on `SIGINT`, stop on `SIGTSTP`).
2.  **Ignore the Signal**: The process can instruct the kernel to simply discard the signal, taking no action. The `SIGKILL` and `SIGSTOP` signals are special exceptions that cannot be ignored or caught.
3.  **Catch the Signal**: The process can register a custom function, called a **signal handler**, to be executed when the signal is received. This allows a program to define its own behavior for handling interruptions.

## Installing Signal Handlers: `signal()` vs. `sigaction()`

There are two primary system calls for setting a signal's disposition:

*   `signal()`: This is the original, simpler C standard library function. However, its use is now strongly discouraged. Its behavior has historically varied across different Unix implementations, making it unreliable for portable and robust applications.
*   `sigaction()`: This is the modern, POSIX-standard function for managing signals. It is more complex but provides a far more reliable, portable, and powerful interface. For any serious systems programming, `sigaction()` is the correct and recommended choice.

A critical consideration when writing signal handlers is the concept of **async-signal-safety**. A signal can arrive at any moment, interrupting the main program's execution at an arbitrary point. If the handler then calls a function that is not reentrant, a race condition can occur, leading to data corruption or a crash. Therefore, only a small subset of system calls and library functions, designated as "async-signal-safe" (like `write()`, but not `printf()`), can be safely called from within a signal handler.

## Signal Handling Logic in a Shell

The logic for handling signals in a shell is nuanced because the shell must act as a manager for other processes. The correct behavior depends on whether the shell itself or a child process is the intended recipient of the signal.

For a simple shell, the logic can be simplified as follows:

*   **When the shell is at the prompt (interactive)**: The shell itself is in the foreground.
    *   It should **catch `SIGINT`**. The handler should not terminate the shell but should instead break out of any pending input read and redisplay the prompt, effectively canceling the current line.
    *   It should **ignore `SIGTSTP`**. There is no concept of "stopping" the shell itself in this context.
*   **When the shell is running a foreground command**: The shell has forked a child and is waiting for it.
    *   When `Ctrl+C` is pressed, both processes receive `SIGINT`. The child process, which has not had its signal handlers modified, will perform the default action and terminate. The parent shell, which has its own custom handler, will ignore the signal or handle it gracefully. The parent's `waitpid()` call will then return, indicating the child was terminated by a signal, and the shell can print a new prompt.

This ensures that user interruptions correctly target the running command, not the shell that launched it, providing the expected interactive experience.
