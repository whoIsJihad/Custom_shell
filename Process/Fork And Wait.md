# `fork()` and `waitpid()`: process creation and reaping from first principles

## What `fork()` actually does in the kernel

When you call `fork()`, the kernel performs one of its most complex operations: **complete process duplication**. To understand what's really happening and why it matters for your shell, you need to know the memory layout and what gets copied.

### A process's address space (before fork)

A running process is not just code. It has several distinct regions in memory:

- **Text segment**: the executable instructions. This is read-only, so the kernel often keeps it shared across multiple instances of the same program.
- **Data segment**: global and static variables that are initialized with non-zero values. For example, `int x = 5;` at file scope ends up here.
- **BSS segment**: uninitialized global and static variables, always zero-filled. The name comes from an obscure IBM assembler directive, but it's meaningful: it's *space* for variables, initialized to zero but not stored in the binary file (it would waste space).
- **Heap**: dynamically allocated memory (everything allocated with `malloc()`). It grows upward, toward higher addresses.
- **Stack**: local variables, function parameters, return addresses. It grows downward, toward lower addresses. Even if two processes share the same codebase, their stacks can be completely different.

Beyond memory, the kernel also maintains per-process structures:

- **File descriptor table**: an array in the kernel (one per process) where `fd_table[0]` is stdin, `fd_table[1]` is stdout, `fd_table[2]` is stderr, etc. Each entry points to a **kernel file description**, which is shared system-wide.
- **Process control block (PCB)**: kernel memory containing the PID, parent PID, signal handlers, CPU registers (saved state), and other metadata.

### How fork copies all of this (with an optimization)

When you call `fork()`, the kernel *conceptually* needs to duplicate all of this. But it does something clever: **copy-on-write (COW)**.

Instead of immediately copying all memory:

1. **Text segment** is truly shared. Both parent and child execute from the same code. The kernel marks the pages as read-only.
2. **Data and BSS** are marked as read-only in both processes. If either tries to *write* to these pages, the CPU raises a page fault. The kernel intercepts this, allocates a new private copy of the page, and lets the write proceed.
3. **Heap and stack** are similarly marked for COW. Most of the time, parent and child don't modify these, so the pages stay shared.
4. **File descriptor table** is immediately shallow-copied. The child gets its own array, but each array entry still points to the same kernel file description. This is critical for pipes.
5. **PCB** includes a new PID (generated by the kernel), a new set of registers, but signal handlers and other attributes are initially the same.

This optimization makes `fork()` very fast. If the child immediately calls `exec()` (which replaces the entire address space), most of those COW pages never get copied. Only if the child modifies memory do copies happen.

### The fork return value: one function, two return values

After `fork()` succeeds, there are now two processes executing the same code at the same point. The *only* difference is the return value:

```c
pid_t pid = fork();
if (pid == 0) {
    /* This code executes in the CHILD. */
    /* Child's fork() returned 0. */
    printf("Child PID: %d\n", getpid());
} else if (pid > 0) {
    /* This code executes in the PARENT. */
    /* Parent's fork() returned the child's PID. */
    printf("Parent created child PID: %d\n", pid);
} else {
    /* pid == -1: fork() failed. No child was created. */
    /* Only parent continues. */
    perror("fork");
}
```

This is why `fork()` is said to return twice: it literally executes the `return` instruction twice, once in each process.

### File descriptors: shared references, not copies

This is the critical piece for understanding pipes and redirection. When a child is born, its FD table is a *copy of the parent's array*, but each FD points to the *same underlying kernel file description*. 

Think of it like this: there's a kernel data structure for "file.txt open for writing at byte offset 1000". The parent has `fd 3` pointing to it. After fork, the child also has `fd 3` pointing to the *exact same* kernel structure. They share the file offset.

This means:
- If the parent reads 100 bytes from `fd 3`, the offset advances to 1100.
- The child then reads from `fd 3`, and it starts from byte 1100, not 0.
- Both reads/writes affect the same file.

This is the root cause of the "don't forget to close unused pipe ends" rule. If the child has `fd 4` open to a pipe write end, and the child doesn't close it, the kernel's reference count for that write end stays at 2 (parent and child both hold it). When the parent tries to read from the pipe, EOF never comes because the child still has a write end open—even if the child isn't actively writing.

## Process termination and why zombies exist

When a child process calls `exit(code)` or `_exit(code)`, or when it receives a signal like SIGKILL, something unintuitive happens: **the process doesn't immediately disappear**.

Instead, the kernel:

1. Releases most of the process's resources: frees its memory, closes its file descriptors, etc.
2. Transitions the process into **zombie state**.

In zombie state:

- The process no longer consumes memory or CPU.
- But the kernel keeps a small entry in the process table with the process's PID and exit status.
- The process is "dead" but its "body" (the process table entry) hasn't been disposed of.

Why does the kernel do this? Because **the parent process owns the right to query the child's exit status**. In Unix, parent-child relationships are meaningful. The kernel preserves the exit status for the parent to collect later.

A zombie hangs around until:

1. The parent calls `wait()` or `waitpid()` to "collect" the exit status (which also deletes the process table entry), **OR**
2. The parent exits, at which point the kernel reassigns the child to PID 1 (`init`), which automatically reaps it.

### Why shells need to care about zombies

If your shell runs a foreground command (`ls`), the shell typically calls `waitpid()` immediately. The child becomes a zombie for a moment, but the parent reaps it right away. No problem.

But if your shell supports background commands (`ls &`), the parent does *not* wait. The child exits, becomes a zombie, and stays in the process table indefinitely—unless the shell explicitly reaps it later. If a user starts many background jobs and the shell never reaps them, the process table fills with zombies, and eventually the system can't create new processes.

Professional shells handle this by either:
- Installing a `SIGCHLD` handler that reaps exited children automatically, or
- Periodically checking for exited background children and reaping them.

## `waitpid()`: the mechanism to collect exit status

The `waitpid()` syscall is how the parent retrieves the child's exit status and cleans up the process table.

### Prototype

```c
#include <sys/types.h>
#include <sys/wait.h>

pid_t waitpid(pid_t pid, int *status, int options);
```

The three parameters:

- **`pid`**: which child(ren) to wait for.
  - `pid > 0`: wait for the child with that specific PID.
  - `pid == -1`: wait for *any* child.
  - (Other values like `pid < -1` handle process groups, more advanced.)

- **`status`**: a pointer to an `int` where the kernel writes an encoded exit status. More on decoding below.

- **`options`**: flags that modify the behavior.
  - `0` (default): **block** the parent until a child changes state (exits, is killed, etc.).
  - `WNOHANG`: **don't block**. Return immediately; if no child has exited, return 0.
  - `WUNTRACED`: also wake up if a child is *stopped* (not just exited). Used for job control.
  - `WCONTINUED`: also wake up if a stopped child has been resumed. Also for job control.

### Return values

- **`> 0`**: A child's PID. A child with that PID changed state. The `status` parameter now contains information about what happened.
- **`0`**: Only possible with `WNOHANG`. No child has changed state; the call returned immediately without blocking.
- **`-1`**: An error. Check `errno`:
  - `ECHILD`: the process has no children, or all children have already been reaped.
  - `EINTR`: the call was interrupted by a signal. The standard response is to retry the call.

### Decoding the status bitfield

The `status` integer is a packed bitfield that the kernel fills in. You must use macros from `<sys/wait.h>` to decode it; the bit layout is not documented for user code:

```c
#include <sys/wait.h>

int status;
pid_t child = waitpid(-1, &status, 0);

if (WIFEXITED(status)) {
    /* Child exited normally, via exit() or return from main(). */
    int exit_code = WEXITSTATUS(status);
    printf("Child exited with code %d\n", exit_code);
} else if (WIFSIGNALED(status)) {
    /* Child was killed by a signal. */
    int signal_num = WTERMSIG(status);
    printf("Child killed by signal %d\n", signal_num);
} else if (WIFSTOPPED(status)) {
    /* Child was stopped by a signal (requires WUNTRACED). */
    int signal_num = WSTOPSIG(status);
    printf("Child stopped by signal %d\n", signal_num);
}
```

For a shell, you usually want to convert the status into a single "exit code" to report to the user or store in `$?`:

```c
static int status_to_exit_code(int status) {
    if (WIFEXITED(status)) {
        return WEXITSTATUS(status);
    }
    if (WIFSIGNALED(status)) {
        /* Convention: 128 + signal number. */
        return 128 + WTERMSIG(status);
    }
    if (WIFSTOPPED(status)) {
        /* For now, your shell probably doesn't handle stopped jobs. */
        return -WSTOPSIG(status);
    }
    return 1;
}
```

### Foreground: fork, exec, and wait immediately

When your shell runs a foreground command, the pattern is:

```c
#include <errno.h>

pid_t child_pid = fork();

if (child_pid == 0) {
    /* Child process. */
    /* Set up any FDs for redirection (see dup2 note). */
    /* Then replace the process image. */
    execvp(argv[0], argv);
    /* If exec returns, it failed. */
    perror("execvp");
    _exit(127);  /* Use _exit to avoid flushing parent's buffers. */
}

if (child_pid == -1) {
    perror("fork");
    return;
}

/* Parent: wait for the child. */
int status;
for (;;) {
    pid_t result = waitpid(child_pid, &status, 0);
    if (result == -1) {
        if (errno == EINTR) {
            /* A signal interrupted waitpid. Retry. */
            continue;
        }
        /* Real error. */
        perror("waitpid");
        break;
    }
    /* Child changed state (exited, killed, etc.). */
    break;
}

int exit_code = status_to_exit_code(status);
printf("Exit status: %d\n", exit_code);
```

The loop is necessary because signals can interrupt `waitpid()`. The standard Unix response is to retry the syscall. (This is an example of "system calls can be interrupted"; in production shells, you see this pattern everywhere.)

### Background: don't wait, reap later

When the shell supports background execution (`cmd &`), the parent does not wait immediately:

```c
pid_t child_pid = fork();
if (child_pid == 0) {
    execvp(argv[0], argv);
    _exit(127);
}
if (child_pid == -1) {
    perror("fork");
    return;
}

/* Don't wait. Just print a message and continue the shell loop. */
printf("Background job %d started\n", child_pid);
/* The child is now a zombie until we reap it. */
```

To reap background jobs later, install a signal handler:

```c
#include <signal.h>

void sigchld_handler(int sig) {
    /* When a child exits, the kernel sends SIGCHLD to the parent. */
    /* Reap all exited children. */
    int status;
    for (;;) {
        pid_t r = waitpid(-1, &status, WNOHANG);
        if (r <= 0) {
            /* No more exited children. */
            break;
        }
        /* Reaped child PID r. */
        printf("Background job %d exited with code %d\n", r, 
               status_to_exit_code(status));
    }
}

int main() {
    signal(SIGCHLD, sigchld_handler);
    /* Now, whenever a child exits, the handler fires and reaps them. */
}
```

Or, if you don't want signal handlers, the shell can periodically call `waitpid(..., WNOHANG)` in its main loop to reap exited background jobs.

## Key insights for your shell

1. **`fork()` is fast because of COW**. The child doesn't get a private copy of memory until it modifies something.
2. **File descriptors are shared after fork**. Both parent and child refer to the same underlying open file. This is why pipes work, and why you must close unused ends.
3. **Zombies exist because the parent owns the exit status**. The kernel preserves it until the parent asks for it with `waitpid()`.
4. **Foreground jobs: wait immediately. Background jobs: wait later** (via handlers or polling).
5. **`EINTR` is everywhere**. Wrap blocking syscalls in a retry loop.
